# アプリストア審査 対応レビュー(Apple Kids カテゴリ / Google Play ファミリー)

2026-07 時点のコードベース(Phaser 3 Web 版)を、将来 Capacitor でストア提出する前提で総点検した結果。
対象規約: Apple App Review Guidelines(特に 1.3 Kids Category / 2.1 完成度 / 5.1 プライバシー)、
Google Play ファミリー ポリシー + Data Safety、COPPA/GDPR-K。

## ✅ すでに審査に強い点(このまま維持する)

| 項目 | 状態 |
|---|---|
| データ収集 | **ゼロ**。通信は自アセットの読み込みのみ。解析SDK・広告SDK・外部フォント・CDNなし(コード検査で確認済み) |
| セーブ | localStorage のみ(端末内)。アカウント不要 → 削除要件・COPPA 同意フローが不要 |
| 広告・課金・通知 | なし(CLAUDE.md の健全設計原則として明文化済み) |
| 外部リンク | ゼロ。Kids カテゴリの「ゲートなし外部誘導禁止」に抵触する箇所なし |
| 失敗ペナルティ | なし(成功保証設計)。過度な射幸性(ガチャ等)なし |
| ライセンス | 地図データの CC BY 4.0 クレジットをアプリ内(せってい)に表記済み |
| オフライン | 完結(Capacitor 化で全アセット同梱可能) |

この「収集ゼロ」構成は両ストアの子供向け審査で最も通りやすい形。**今後も解析SDKを安易に足さない**こと。

## 🔴 High: 提出前に必ず直す(コード対応)

### H1. 保護者ゲートが弱すぎる+実装が window.prompt ✅対応済み
> ゲーム内モーダル+テンキーの `ui/parentGate.ts` に置換。2桁×1桁の掛け算(12〜49 × 3〜9)、
> 漢字表記で子供に読めない、3回誤答で閉じる。window.prompt / confirm は全廃。
- 現状: 1桁の足し算(2〜9+2〜9)を `window.prompt` / `window.confirm` で出題
- 問題: Apple の Kids カテゴリ運用では「子供が容易に突破できないゲート」が求められる。
  1桁足し算は対象年齢(6-12)の子が普通に解ける。また prompt はネイティブでは
  OSダイアログになり、体裁・ローカライズの面でも弱い
- 対応: ゲーム内UIのゲートモーダルに置き換え、難度を大人向けに
  (例: 2桁×1桁の掛け算/「じゅうさん たす にじゅうろく は?」のように
  かな書き数字で出題/「3びょう ながおし してから こたえる」の組み合わせ)

### H2. 管理者機能(⏩まんたん)が子供に露出 ✅対応済み
> せってい画面から管理者ボタンを撤去し、「保護者メニュー」(H1のゲートの先)へ移動。
> iPad での動作確認用に保護者は引き続き使える。__mqAdmin の window API は E2E 用に残置。
- 現状: せってい画面の先頭に誰でも押せる形で表示
- 問題: デバッグ機能の本番露出は Apple 2.2(ベータ/デモ)と見なされるリスク+ゲーム性を壊す
- 対応: `import.meta.env.DEV`(または URL パラメータ)でのみ表示する。
  E2E は preview ビルドを使うため、テストフック(__mqAdmin)は window API として残してよい
  (UIに出さなければ審査対象にならない)

### H3. 「じゅんびちゅう」文言の世界観化 ✅対応済み
> 全て雲の世界観に置換: 「まだ もやもやぐもの なか」「くもの むこう」等。🚧絵文字も撤去。
- 現状: にほんぜんこくの7地方に「じゅんびちゅう」表示
- 問題: Apple 2.1(完成度)は「Coming Soon」区画に厳しい。開発都合の文言は
  未完成アプリと judged されるリスク
- 対応: ストーリー文言に置き換える(例: 「まだ もやもやぐもの なか…」)。
  雲に閉ざされている=ゲーム世界の状態、として自然化できる(ストーリー実装済みなので低コスト)

### H4. プライバシーポリシーの用意 ✅対応済み(日本語版)
> /project01/privacy.html を公開(public/privacy.html)+アプリ内の保護者メニューにも同内容を表示。
> 英語版と問い合わせ先の記載はストア提出時に追加。
- 両ストアとも URL 必須(子供向けは特に厳格)。「なにも収集しない」内容で1ページ作り、
  GitHub Pages に置く(例: /project01/privacy.html)。日本語+できれば英語
- 申告フォーム: Apple「App Privacy(栄養表示)」= データ収集なし /
  Google「Data Safety」= 収集・共有なし、で一貫させる

### H5. トリビア・クイズの裏取り完了
- data の `check` フラグ付き(統計・発祥・祭り名など十数件)を提出前に全て裏取りし、
  フラグを外すか文言を弱める。教育系で事実誤認は審査より先に信頼問題

### H6. アプリ名の確定と商標確認
- 「めいさんクエスト(仮)」の (仮) を外す前に、商標・既存アプリ名との衝突を確認
  (「〜クエスト」は既存ゲーム商標が多い)。ストア表示名・アイコン・スクショ一式も必要

## 🟡 Medium: 提出作業とセットで対応

- **M1. 年齢帯の選択**: Apple Kids は「5歳以下 / 6–8 / 9–11」から1つ。対象6-12なら
  「6–8」か「9–11」を選ぶ。GAME_SPEC の「4〜8歳」記述も正式に更新する(要ユーザー確認)
- **M2. iPad/タブレットの見栄え**: 480×800 固定 FIT のレターボックス。動作は問題ないが
  Kids アプリは iPad 利用が主流で、審査には iPad スクショも必要。最低限レターボックス部の
  背景を馴染ませる(現状 #dfeef4 単色なので実質OK。実機で要確認)
- **M3. セーブの堅牢化**: localStorage は「WebViewデータ消去」「ストレージ圧迫時のOS削除」で
  消えることがある。Capacitor 化の際に Preferences/ファイル保存へ移行(審査要件ではないが
  「データが消えた」低評価はKidsアプリで致命的)
- **M4. Capacitor 化の技術タスク**: セーフエリア(ノッチ)対応/Android 戻るボタンの
  ハンドリング/スプラッシュ・アイコン生成/iOS ビルド用の Mac 確保
- **M5. レーティング質問票**: Google は IARC 質問票、Apple は年齢レーティング設定。
  暴力・課金・UGC・位置情報すべて「なし」で回答できる構成なので作業のみ
- **M6. ミュート既定の検討**: ROADMAP 記載の「ミュート既定・保護者配慮」。現状は音アリ既定+
  🔊トグル。審査要件ではないが Kids アプリの作法として起動時音量は控えめに(BGMは対応済み)

## 🟢 Low: 将来フリーミアム(エリア解放課金)にする時の注意

- 課金導線は**保護者ゲートの内側**に置く。子供に直接「買って」と促す文言・演出は禁止
  (「おうちのひとと いっしょに みてね」方式)
- Kids カテゴリではサードパーティ広告/解析SDKは実質禁止。課金レシート検証も
  自前 or ストア標準のみで設計する
- 「購入の復元(Restore Purchases)」実装が必須
- 価格・解放範囲を子供が誤解しない表示(「かんとうは ずっと むりょう」等の明示)
